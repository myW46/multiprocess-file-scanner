# Система многопроцессного анализа файлов на вредоносные паттерны

Данный проект представляет собой комплекс из трех консольных утилит для поиска потенциально опасного содержимого в файлах.

## Архитектура системы

Система разделена на три независимых компонента, взаимодействующих через сетевые сокеты и IPC:

1.  **Сервер (`/server`)** — Ядро системы.
    - Для каждого нового клиента создается отдельный дочерний процесс. Это изолирует обработку файлов и повышает общую отказоустойчивость.
    - Дочерние процессы агрегируют статистику найденных угроз в общем сегменте памяти, доступном основному серверу.
    - Поиск паттернов реализован на базе конечного автомата. Это гарантирует сканирование файла за один проход ($O(N)$) независимо от количества сигнатур в базе.
    - При завершении работы сервер корректно дожидается закрытия всех дочерних процессов и освобождает системные ресурсы.

2.  **Клиент (`/client`)**
    - Обеспечивает передачу целевого файла на сервер через TCP-сокеты.
    - Принимает и выводит результаты анализа (обнаруженные сигнатуры, их описание и тип).

3.  **Утилита статистики (`/stats`)**
    - Подключается к серверу через **Named Pipe**.
    - Запрашивает и выводит актуальную статистику по всем просканированным файлам к этому моменту.

## Конфигурация и база сигнатур

База сигнатур хранится в файле `config.json` в корневой директории. Поддерживается два режима детекции:
* **text**: Поиск текстовых строк.
* **hex**: Поиск бинарных последовательностей. HEX-строки автоматически транслируются в бинарный вид перед загрузкой в автомат.

Для работы с конфигурацией используется библиотека `JSON for Modern C++`.

## Модульное тестирование (Unit Testing)

Для верификации поискового движка в проект интегрирована система тестов на базе фреймворка **doctest**. 

### Что проверяют тесты:
- Алгоритмическая точность**: Построение суффиксных ссылок в боре и корректность переходов автомата.
- Корректная обработка нулевых байтов (`0x00`) и значений вне диапазона ASCII, что критично для анализа исполняемых файлов.
- Проверка на отсутствие ложноположительных срабатываний на чистых данных.
- Способность алгоритма находить вложенные паттерны (например, когда одна сигнатура является частью другой).

## Система сборки (Build System)

Проект использует иерархическую систему сборки на базе **CMake (версия 3.17)**.

### Архитектура CMake
В корневой директории расположен основной файл `CMakeLists.txt`, который координирует сборку всей системы:
- Через команду `add_subdirectory()` подключаются независимые модули (сервер, клиент, статистика, тесты).
- Задаются единые стандарты компиляции (C++17).
